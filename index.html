未命名

发表于 2018-06-16   |  

title: golang中的channel挖掘
date: 2016-06-16 21:07:00
tags: [golang]

当我刚开始接触golang中的channel时，我将其看作为一个类似于队列的数据结构,提供goroutine之间的自动化同步访问。这种结构上的理解让我编写了许多复杂的并发代码。
慢慢地，我开始忘掉channel的数据结构，而将其理解为信号的行为，即一个channel允许goroutine向另一个goroutine发出关于某些特定时间的信号。从而，我的理解是，信号是使用channel做一切工作的核心，将channel的行为作为信号机制来考虑将有利于编写具有良好定义和精确行为的代码。

下面将从channel的三个关键属性来理解channel的信号机制是如何工作的：

  • 交付确认
  • 状态
  • 是否含有数据
    以上三个属性共同实现可channel的关于信号的设计理念。下面将结合图示和代码分别对这三个属性进行分析。

交付确认

属性”交付确认“可以回答问题”我需要确认一个特定的goroutine发送的信号是否已经被收到“

go func() {
    p := <-ch    //Receive
}()

ch <- "paper"    //send

发送的goroutine（05行）是否需要接收的goroutine（02行）对paper信号进行确认？
基于该问题，我们就可以理解golang中提供的两种不同类型channel的差别：Unbuffered（无缓冲）和Buffered(带缓冲的)，他们针对“交付确认”这一属性提供了不同的表现。

状态

channel的表现受其当前的状态所决定，channel可能处于的状态包括nil，open和closed。
下面的代码实例中展示了三种状态的描述：

var ch chan string            // A channel is in a nil state when it is declared to its zero value

ch = nil                     // A channel can be placed in a nil state by setting it to a nil

ch := make(chan string)        // A channel is in a open state when it's made using the built-in function make

close(ch)                    // A channel is in a closed state when it's closed using the built-in function close

channel所处于的状态决定了对其进行send和receive操作的表现（如前文所述，我们将channel理解为信号的行为，因为将操作称为send和receive，而不说read/write）。如下图所示：

当一个channel处于nil状态时，任何试图对channel进行的send/receive操作都将被阻塞；
当一个channel处于open状态时，信号可以在channel中被发送和接收；
当一个channel处于closed状态时，不能再发送信号，但仍可能接收信号。

是否含有数据

最后一个需要考虑的信号属性是是否需要携带数据。

ch <- "paper"

当你的信号中携带数据时，通常发生在：
1）一个goroutine需要开启一项新任务；
2）一个goroutine需要反馈结果。

close(ch)

当你的信号中不携带数据时，通常发生在：
1）一个goroutine需要被停止；
3）一个goroutine需要反馈它已经完成，并不需要反馈结果
2）一个goroutine需要报告它已经完成或关闭了处理。

当然，这些总结规则会有些例外，但这可以针对大部分场景。因此，本文中将对其进行重点讨论。

没有数据的信号的一个好处是单个goroutine可以同时发出多个信号，而带数据的信号在goroutine之间始终是1对1的交换。

带数据的信号

当你要使用带数据的channel时，你可以根据需要得到的确认类型来选择channel配置选项，这三个配置选项分别是Unbuffered，Buffered>1和Buffered=1，如下图所示。

  • Unbuffered channel提供了一个确认，即确认接收到的信号已被接收，因为信号的接收发生在信号发送完成之前；
  • Buffered>1 channel不能保证发送的信号已经被接收，因为信号的发送在信号的接收完成之前完成；
  • Buffered=1 channel提供了延迟确认，它可以保证先前发送的信号已经被接收，因为第一个信号的接收发生在第二个信号的发送完成之前。

不带数据的信号

不带数据的信号主要预留为取消操作.它允许一个goroutine向另外的goroutine发出信号停止或继续它们正在做的事情.该操作可以通过Buffered和UnBuffered两种channel实现，但相对来说使用Buffered channel会更常见一些。

不带数据信号操作的一个常见例子是内置函数close。如前文所述，你仍然可以在关闭的信道上接收信号。事实上，在关闭的channel上的任何接收操作都不会阻塞，并且总是会返回。

实际应用场景

在对channel的属性进行了总结之后，进一步理解他们的最好方式就是通过实际代码中的一些场景。下面通过几个例子来加深对channel的认识，并且在例子中，将goroutine想象成人。

带数据-确认-无缓冲 Channel

  • 场景1 等待接收任务
    想象一位经理雇佣一位新员工的场景。该场景下，你希望新雇员完成一些工作，但需要等到您准备好为止。这是因为你需要在他们开始之前递给他纸。

    01 func waitForTask() {
    02 ch := make(chan string)
    03
    04 go func() {
    05 p := <-ch
    06
    07 //雇员开始工作
    08
    09 //雇员完成工作后离开
    10 }()
    11 time.Sleep(time.Duration(rand.Intn(500)) * time.Millisecond)
    12 ch <- “paper”
    13}

在第02行，程序创建了一个带string数据的Unbuffered channel，然后在第04行，新雇员在工作之前被告知在05行等待开始的信号，第05行是接收信号，这将使得新雇员在等待纸时阻塞。一旦新雇员收到你递过去的纸，他就可以开始工作，并在完成之后离开。
你作为经理与新雇员同时在工作，所以你在雇佣了04行的新员工之后，需要一些额外的事情来完成雇佣流程上的事情。最后，当一切就绪，你向新雇员发送信号。在第12行，发出带数据的信号，数据就是“paper”。由于使用的是Unbuffered channel，可以保证一旦发送操作完成，新雇员就以及收到了“paper”。

  • 场景2 等待接收结果
    在接下来这个场景中，情况发生了变化。这一次，你希望新雇员在被雇佣时立即执行一项任务，你需要等待他们工作的结果，待他完成之后，才能继续你的工作。

    01 func waitForResult() {
    02 ch := make(chan string)
    03
    04 go func() {
    05 time.Sleep(time.Duration(rand.Intn(500)) * time.Millisecond)
    06
    07 ch <- “paper”
    08 //雇员完成工作
    09 }()
    10 p := <-ch
    11}
    在第02行，程序创建了一个带string数据的Unbuffered channel，然后在第04行，新员工被雇佣并立即投入工作。当你在04行新员工被雇佣之后，你在第12行等待着其工作结果。
    一旦工作在05行完成，通过执行数据发送的channel在07行告知你。由于这是Unbuffered channel，接收发生在发送完成之前，因此新雇员保证您已经接收到结果。在这种场景下，你不知道新雇员需要多长时间才能完成工作。

  • 开销
    Unbuffered channel提供了对接收到的信号的保证。这很棒，但该保证的成本是未知的时延。即在“等待接收任务”场景中，新雇员不知道你要多久才能发送文件；而在“等待接收结果”场景中，你不知道新雇员要多久才能把结果发送给你。
    在这两种情况下，因为我们需要交付确认，所以要忍受未知的时延。反之则不需要。

带数据-无确认-Buffered>1 Channel

当你不需要知道被发送的信号是否被接收的时候，考虑以下两种场景：Fan Out和Drop。
带Buffered的channel具有可用于存储发送数据的空间，那么如何决定所需要的空间呢？

  • Fan Out
    Fan Out模式允许你让多个员工同时工作于一个问题。因为如果你让每个员工完成一个任务，那么将会收到很多报告，你可以确保你的盒子有足够的空间来容纳这么多的报告。这是员工不需要等待你而提交报告的好处。
    想象一下你还是经理，但这次需要雇佣一个员工团队。你希望每个员工都执行一个单独的任务，当每个员工完成他们的任务时，他们需要为你提供一份报告，并放置于你的办公桌上。

    01 func fanOut() {
    02 emps := 20
    03 ch := make(chan string, emps)
    04
    05 for e:=0; e<emps; e++ {
    06 go func() {
    07 time.Sleep(time.Duration(rand.Intn(200)) * time.Millisecond)
    08 ch <- 10="" 11="" 12="" "paper"="" 09="" }()="" }="" for="" emps=""> 0 {
    13 p := <-ch
    14 fmt.Println(p)
    15 emps–
    16 }
    17 }
    在第03行，程序创建了一个带string数据的Buffered channel，这一次，该channel的缓冲区大小设定为20。
    在第05行至10行之间，20名新员工被雇佣兵立即开始工作。你不知道每位员工完成任务会需要多少时间（第07行）。然后在08行，员工完成任务后发送报告，但该发送过程并不阻塞等待接收。因为盒子里每个员工都有空间，channel上的发送只与其他在同一时间发送报告的员工竞争。
    12行至16行之间的代码表示你等待所有20名新雇员完成他们的工作并发送报告。一旦受到报告，将在第14行打印，并且本地计数器变量被递减以指示雇员已完成。

  • Drop
    Drop模式允许你在雇员有能力的时候放弃一些工作。这有利于你继续接受来自客户的工作，并且从不在接受工作时施加压力后延迟。关键是要知道你什么时候才真正有能力，这样你就不会对你应该完成的工作超时。通常需要集成测试或度量来帮助你识别这个数字。

想象一下你仍然是经理，雇佣一名员工完成工作。你有一个希望员工执行的单独任务。当员工完成任务时，你不想知道他们完成了任务，重要的是你能不能把新工作放在箱子里。如果你不能执行发送，那么你知道你的箱子里已经满了，而雇员在工作。在这时，新的工作需要被丢弃。

01 func selectDrop() {
02    const cap = 5
03    ch := make(chan string, cap)
04    
05    go func() {
06        for p := range ch {
07            fmt.Println("employee : received :", p)
08        }
09    }() 
10    
11    const work = 20
12    for w:=0; w<work; w++ {
13        select {
14            case ch <- "paper":
15                fmt.Println("manager : send ack")
16            default:
17                fmt.Println("manager : drop")
18        }
20    }
21    close(ch)
22 }

在第03行，程序创建了一个带string数据的Buffered channel，这一次，该channel的缓冲区大小设定为5。
在第05行至09行之间，雇佣一名新员工来处理这项工作。for range用于信道接收的范围，每次收到paper，就在第07行处理。
在第11到第20行之间，你尝试向员工发送20张paper，这一次使用SELECT语句来实现，如果缓冲区没有更多的空间（default），发送将被阻塞，在第17行放弃发送。
最后，在第21行，内置函数close被调用关闭channel。这将在没有数据的情况下向完成工作的员工发出信号。

  • Cost
    一个Buffered>1的缓冲channel并不能保证发送的信号被接收。这样带来了一个好处，就是在两个goroutine的通信之间减少了时延。在Fan Out场景中，每个发送报告的雇员都有一个缓冲区，而在Drop场景中，如果缓冲区的容量到达，新到达的工作将会被丢弃以保证现有的进行。
    在这两种选择中，缺乏确认是我们必须面对的，因为延迟的减少更为重要。最小到零的延迟要求不会对系统的整体逻辑造成问题。

带数据-延迟确认-带缓冲Channel

当需要知道在发送信号之前是否接收到发送的前一个信号时，等待接收任务场景开始发挥作用。

  • 场景1 等待接收任务
    在这种场景下，你有一些新员工，但他们将不仅仅完成一项任务。你要一个接一个地为他们提供许多任务。但是，他们必须先完成每项任务，然后才能开始新任务。由于他们一次只能处理一项任务，所以在切换工作之间存在延迟问题。如果需要在不丢失雇员可以开始下一项工作的信号情况下，同时降低延迟，这种情况会起到作用。
    这是该channel的好处。如果你和雇员都在按照预定的节奏进行，那么你们不需要互相等待。每次你发送一张纸时，缓冲区都是空的。每次你的雇员要有更多的工作，缓缓区就满了。这是一个完美的工作流程对称性。
    最好的part就是这样。如果任何时候你试图发送一张纸却不能，因为缓冲区是满的，那你知道你的员工有问题并且停下来查看。这是延迟确认的所在。当缓冲区为空并且你执行发送，你可以保证你的员工已经完成了你发送的最后一份工作。如果你执行发送却不能，你可以确认他们没有完成。

    01 func waitForTasks() {
    02 ch := make(chan string, 1)
    03
    04 go func() {
    05 for p := range ch {
    06 fmt.Println(“employee : working :”, p)
    07 }
    08 }()
    09
    10 const work = 10
    11 for w:=0; w<work; w++ {
    12 ch <- “paper”
    13 }
    14
    15 close(ch)
    16 }
    在第02行，程序创建了一个带string数据的大小为1的Buffered channel，在第04行至08行之间，一个新员工被雇佣来处理工作，for range用于channel接收信号。每次收到paper，就在第06行处理。
    在第10行至第13行之间，你开始向员工发送任务，如果你的员工可以尽可能快的速度运行，那么你俩之间的等待时间就会减少，但是每一次发送成功，你都能保证你提交的最后一份工作正在被处理。
    最后，在第15行中，内置函数被调用关闭channel。这将向员工发出信号他们已经完成了工作。但是，在for range终止之前，你提交的最后一项工作将被接收（刷新）。
